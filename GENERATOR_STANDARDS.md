# NAT 2.0 Generator Standards

This file is the **single source of truth** for the NAT 2.0 code generator.
Every rule below applies to **every customer site** generated by the platform.
No rule is site-specific. The word "customer-site" is a placeholder for any domain.

---

## Rule 1 — Method Name Contract (Layer N → N+1 Threading)

Every method name called in a business action file **MUST** exactly match a method
defined and exported from the referenced POM class.

The generator resolves method names by **reading the actual POM output** it produced —
not from assumptions, training memory, or naming patterns.

**Implementation contract:**
After generating `pages/XxxPage.ts` (Layer 2), the generator extracts the list of all
`async methodName()` declarations and passes that exact list as context to the next
Claude call that generates `actions/business/xxx.actions.ts` (Layer 4).

```
Layer 2 output → extract [ methodName, ... ] → inject into Layer 4 prompt
Layer 4 output → extract [ functionName, ... ] → inject into Layer 5 prompt
```

❌ WRONG (invented):  `await pg.navigateToServices()`
✅ CORRECT (from POM): `await pg.clickServicesNavLink()`

---

## Rule 2 — No Phantom Imports

Never generate `import { X } from 'path/to/Y'` unless file Y is either:
- part of the current generation output batch, OR
- a file known to already exist in the project

If a test imports `prepareSite` from `../helpers/universal`, then
`helpers/universal.ts` **must be emitted in the same generation batch**.

The generator always emits `helpers/universal.ts` whenever it generates a test file.

---

## Rule 3 — No Hardcoded Absolute URLs in POMs

POM `goto()` calls **always** use relative paths:

```typescript
// ✅ CORRECT — environment-agnostic
await this.page.goto('/services/agile-development');

// ❌ WRONG — breaks for dev/staging/prod/any-customer
await this.page.goto('https://www.customer-site.com/services/agile-development');
```

Playwright resolves relative paths against `baseURL` in `playwright.config.ts`.
This makes the framework work across dev / staging / prod without changes.

The generator **never** emits absolute `https://` URLs inside any POM class body.

---

## Rule 4 — No Assertions in the POM Layer (Layer 2)

Page Object Model classes interact with the DOM only: locate, click, fill,
scroll, hover, read. They **never** import or call `expect()`.

```typescript
// ✅ CORRECT — POM returns the value; caller asserts
async getHeadingText(): Promise<string> {
  const loc = SitePageLocators.heading(this.page);
  await loc.waitFor({ state: 'visible' });
  return (await loc.textContent()) ?? '';
}

// ❌ WRONG — assertion inside POM
async verifyHeading(): Promise<void> {
  const text = await loc.textContent();
  expect(text).toBe('Expected heading');  // ← belongs in Layer 4
}
```

Assertions belong exclusively in:
- **Layer 4** (business actions) using `verifyText()`, `verifyUrl()`, `verifyVisible()`
- **Layer 5** (test files) for final outcome checks

The generated POM **never** imports `expect` from `@playwright/test`.

---

## Rule 5 — `verifyText` vs `verifyVisible`: Strict Usage Contract

| Function | Input type | Used for |
|---|---|---|
| `verifyText(page, string)` | Prose string | Visible text, headings, labels, button copy |
| `verifyVisible(page, selector)` | CSS selector | Element existence by `[data-testid]`, `.class`, `#id` |

```typescript
// ✅ CORRECT
await verifyText(page, 'Custom accelerators, assured results');
await verifyVisible(page, '[data-testid="submit-btn"]');

// ❌ WRONG — text string passed to selector function
await verifyVisible(page, 'Custom accelerators, assured results');

// ❌ WRONG — CSS selector passed to text function
await verifyText(page, '[data-testid="submit-btn"]');
```

---

## Rule 6 — `verifyUrl` Always Receives a String Path

```typescript
// ✅ CORRECT
await verifyUrl(page, '/data-visualization');

// ❌ WRONG — RegExp literal is not the function signature
await verifyUrl(page, /data-visualization/);
```

The `verifyUrl` function signature is `(page: Page, path: string)`.
The function internally converts the path to a regex for matching.
The generator never emits a RegExp literal as the `verifyUrl` argument.

---

## Rule 7 — XPath Href Locators Must Use `contains()`, Never Exact Match

```xpath
<!-- ✅ CORRECT — works for relative, absolute, any environment -->
//a[contains(@href,"data-visualization") and contains(normalize-space(text()),"Learn More")]

<!-- ❌ WRONG — bare relative href never matches the DOM value -->
//a[@href="competency/data-visualization"]

<!-- ❌ WRONG — absolute href breaks across environments -->
//a[@href="https://www.customer-site.com/data-visualization"]
```

Always use `contains(@href, 'distinguishing-path-segment')`.

---

## Rule 8 — Tab and Role Locators Must Use Semantic Content

```xpath
<!-- ✅ CORRECT — survives tab reorder or count changes -->
//button[@role="tab" and contains(normalize-space(.),"Digital Product")]

<!-- ❌ WRONG — breaks when any tab is added, removed, or reordered -->
//button[@role="tab" and @aria-label="1 of 6"]
```

Always identify tabs by their **visible label** or **semantic aria-label**
that describes the content, not the position.

---

## Rule 9 — Every POM File Must Have a Corresponding Locators File

For each `pages/XxxPage.ts`, there must be a `locators/XxxPage.locators.ts`.
The POM imports all locators **exclusively** from its paired locators file.

No raw XPath, CSS, or selector strings appear anywhere inside a POM class body.

```typescript
// ✅ CORRECT — POM delegates to locators file
async clickSubmit(): Promise<void> {
  const loc = SitePageLocators.submitButton(this.page);
  await loc.waitFor({ state: 'visible' });
  await loc.click();
}

// ❌ WRONG — inline selector in POM
async clickSubmit(): Promise<void> {
  await this.page.locator('xpath=//button[text()="Submit"]').click();
}
```

---

## Rule 10 — Every POM Used in Business Actions Must Be Instantiated Correctly

If a business action **navigates to Page X and interacts with Page X**, it
instantiates the POM **for Page X** — not the homepage POM.

```typescript
// ✅ CORRECT — uses the POM for the page being tested
export async function reviewDataVisualization(page: Page): Promise<void> {
  const pg = new DataVisualizationPage(page);
  await pg.navigateToPage();
  ...
}

// ❌ WRONG — uses homepage POM while operating on data-viz page
export async function reviewDataVisualization(page: Page): Promise<void> {
  const pg = new HomePage(page);  // wrong POM!
  ...
}
```

---

## Rule 11 — Tests Must Be Atomic: One Test Per User Scenario

Each `test()` block covers exactly **one** testable scenario.

```typescript
// ✅ CORRECT — each test is independently runnable
test('Home page loads and displays key content', async ({ page }) => { ... });
test('Navigating to Data Visualization page succeeds', async ({ page }) => { ... });
test('Capability sections are all visible', async ({ page }) => { ... });

// ❌ WRONG — chaining multiple unrelated scenarios into one test
test('All the things', async ({ page }) => {
  await verifyHomePage(page);
  await navigateToDataViz(page);
  await verifyAllSections(page);  // if navigation fails, this is never reached
});
```

---

## Rule 12 — Test Files Must Not Contain Locators or Selectors

Test files (Layer 5) call only:
- Business action functions (Layer 4)
- Generic assert functions from Layer 3 for **final state** checks
- `navigateTo()` for page load
- `prepareSite()` from `helpers/universal`

```typescript
// ❌ WRONG — XPath in test file
await page.locator('xpath=//button[@data-testid="submit"]').click();

// ❌ WRONG — CSS selector assertion in test file
await verifyVisible(page, '[data-testid="result-panel"]');
// (unless this selector exists in the project's locators repository)
```

---

## Rule 13 — `prepareSite()` Is the Universal Site-Readiness Gate

Every test follows this exact opening sequence:

```typescript
await navigateTo(page, testData.baseUrl);   // or specific page URL
await prepareSite(page);                    // dismiss banners, wait for networkidle
```

`prepareSite()` handles cookie/consent banners, popups, and load state for **any**
customer site. It silently continues if no banners are found.

---

## Rule 14 — No Unused Parameters or Imports

```typescript
// ✅ CORRECT
test('Home page loads', async ({ page }) => { ... });

// ❌ WRONG — `context` declared but never used
test('Home page loads', async ({ page, context }) => { ... });
```

TypeScript strict mode warns on unused variables.
The generator emits only parameters that are referenced in the test body.

---

## Rule 15 — `baseURL` Is Always Environment-Driven

```typescript
// ✅ CORRECT — playwright.config.ts
baseURL: process.env.BASE_URL || 'https://www.customer-default-url.com'

// ✅ CORRECT — fixtures/test-data.ts
baseUrl: process.env.BASE_URL || 'https://www.customer-default-url.com'
```

The fallback default is the customer's **production URL**.
Never hardcode any URL in config files except as the env-var fallback.

The `.env.example` file always documents:
```
BASE_URL=https://www.customer-site.com
TEST_USERNAME=
TEST_PASSWORD=
```

---

## Rule 16 — TypeScript Identifiers Must Not Contain Hyphens

Class names, variable names, and exported constants are valid TypeScript identifiers.

```typescript
// ✅ CORRECT
export class DataVisualizationPage { ... }
export const DataVisualizationPageLocators = { ... };

// ❌ WRONG — hyphens in identifiers are invalid TypeScript
export class Data-visualizationPage { ... }
export const Data-visualizationPageLocators = { ... };
```

**File names may contain hyphens** (e.g. `Data-visualizationPage.ts`).
**Class names and variable names must not.**

Page names derived from URL slugs that contain hyphens must be converted to PascalCase:
`data-visualization` → `DataVisualization`
`agile-development` → `AgileDevelopment`

---

## Rule 17 — Never Capture reCAPTCHA or Third-Party iFrame URLs as Test Steps

The recorder may capture navigation events to Google reCAPTCHA iframes:
```
https://www.google.com/recaptcha/api2/anchor?ar=1&k=...
```

These are **noise** — they are not user interactions and must never be emitted
as `page.waitForURL()` calls in any generated file.

Filter pattern (discard any step whose URL matches):
```
/recaptcha|\.google\.com|\.doubleclick\.net|\.googletagmanager\.com|\.analytics|\.hotjar\./
```

## Validation Checklist (run after every generation)

```
1.  npx tsc --noEmit                         → 0 errors
2.  grep -r "import.*expect.*playwright" pages/   → 0 matches (no assertions in POMs)
3.  grep -r 'goto.*https://' pages/          → 0 matches (no absolute URLs in POMs)
4.  grep -r '@href="[^h]' locators/          → 0 matches (no bare relative hrefs)
5.  grep -rn "xpath=\|\[@" tests/ | grep -v "^.*://"  → 0 executable matches
6.  grep "fullyParallel" playwright.config.ts → fullyParallel: true
7.  grep "prepareSite" helpers/universal.ts   → function exists and exported
8.  npx playwright test --list                → N tests listed, 0 errors
```
