PHI Detected — Bypass Path
When PHI is found at gate ② → both AI Enrichment (③) and AI Routing (④) LLM calls are skipped entirely. Routing falls back to keyword-only rules using subject only (body is withheld from all processing). Ticket is created and flagged ContainsSuspectedPhi = true. EventId 1001 (BLOCKED) is emitted; agent is prompted to review manually.

PHI Patterns Detected (8)

SSN MRN DOB (labeled) DOB (bare date) Phone Email Addr NPI ZIP+4
  • Source-generated [GeneratedRegex] — zero allocation
  • Conservative (high recall) — false positives preferred over misses
  • Scans subject and body, not just body
  • Returns redacted sample — never logs full PHI value

Audit EventIds

  • 1001 PHI_GATE_BLOCKED — enrichment blocked, PHI detected
  • 1002 PHI_GATE_PASSED — cleared gate, proceeding to LLM
  • 1003 ROUTING_PHI_SKIP — routing LLM skipped due to PHI flag
  • 1004 ROUTING_LLM_CALLED — routing LLM initiated (length only, no content)
  • Query compliance: WHERE EventId = 1001 in App Insights / Seq
  • Azure SDK capped at Warning — no HTTP bodies in logs

Feature Flags (3)

  • AI_EMAIL_ENRICHMENT — controls LLM enrichment call
  • AI_DEPARTMENT_ROUTING — controls LLM routing call; keyword rules always available as fallback
  • AI_PHI_SCANNING — PHI scanner runs regardless; flag gates whether to block vs warn
  • DB-backed · 30s IMemoryCache TTL · No deployment needed to toggle
  • Admin UI at /api/admin/feature-flags — role-gated

BAA Compliance Controls

  • Azure OpenAI endpoint only — never OpenAI.com direct
  • ApiKey in env var / Azure Key Vault — never in source control
  • appsettings.json caps Azure.* SDK at Warning
  • System prompt: "NEVER include patient names, SSNs, DOBs, MRNs"
  • LLM output validated — PHI-like content in response causes confidence=0
  • NullLlmProvider when credentials absent — safe local dev default

Routing Decision Tree

  • PHI detected → keyword rules on subject only (body withheld)
  • AI_DEPT_ROUTING flag off → keyword rules on full text
  • LLM confidence < 0.75 → keyword fallback
  • LLM returns unknown departmentId → keyword fallback
  • No keyword match → first department (catch-all, conf=0.3)
  • Source tagged: AI / Rule / TicketTypeDefault

Resilience & Degradation

  • All LLM calls wrapped in try/catch — exception returns empty result, never throws
  • NullLlmProvider — drop-in replacement for zero-config environments
  • MockLlmProvider — deterministic CI responses via SetNextResponse()
  • Feature flags DB-backed — disable AI instantly without redeploy
  • Routing always has keyword fallback — ticket always gets a department
  • Email intake returns 200 OK even on AI failure — no webhook retries
